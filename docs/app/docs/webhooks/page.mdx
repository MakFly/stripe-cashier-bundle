# Webhooks

## Configuration

Stripe envoie des événements à votre endpoint webhook. Configurez le secret de signature dans `config/packages/cashier.yaml` :

```yaml
cashier:
    webhook:
        secret: '%env(STRIPE_WEBHOOK_SECRET)%'
        tolerance: 300  # tolérance en secondes (défaut : 300)
        events:
            - customer.subscription.created
            - customer.subscription.updated
            - customer.subscription.deleted
            - invoice.paid
            - invoice.payment_failed
            - invoice.payment_action_required
            - checkout.session.completed
            - customer.updated
            - customer.deleted
            - payment_method.updated
```

## Route

Le bundle expose automatiquement le endpoint :

```
POST /cashier/webhook
```

Assurez-vous d'avoir importé la route dans `config/routes.yaml` :

```yaml
cashier_webhooks:
    resource: '@CashierBundle/Resources/config/routes/webhook.yaml'
```

Enregistrez cette URL dans votre [Dashboard Stripe → Webhooks](https://dashboard.stripe.com/webhooks).

|-----------------|---------|
| `customer.subscription.created` | `SubscriptionCreatedHandler` |

## Créer un handler personnalisé

Implémentez `WebhookHandlerInterface` et taguez le service `cashier.webhook_handler` :

```php
use CashierBundle\Contract\WebhookHandlerInterface;
use Stripe\Event;
use Symfony\Component\DependencyInjection\Attribute\AutoconfigureTag;

#[AutoconfigureTag('cashier.webhook_handler')]
final readonly class MyCustomHandler implements WebhookHandlerInterface
{
    public function handles(): array
    {
        return ['payment_intent.succeeded'];
    }

    public function handle(Event $event): void
    {
        $paymentIntent = $event->data->object;
        // Votre logique...
    }
}
```

Un handler peut gérer **plusieurs types d'événements** en retournant plusieurs entrées dans `handles()`.

Vous pouvez aussi étendre `AbstractWebhookHandler` pour accéder aux helpers :

```php
use CashierBundle\Webhook\Handler\AbstractWebhookHandler;
use Stripe\Event;

#[AutoconfigureTag('cashier.webhook_handler')]
final readonly class MyHandler extends AbstractWebhookHandler
{
    public function handles(): array
    {
        return ['customer.subscription.updated'];
    }

    public function handle(Event $event): void
    {
        $subscription = $this->getStripeSubscription($event);
        $customer = $this->getStripeCustomer($event);
        $invoice = $this->getStripeInvoice($event);
    }
}
```

## Événements Symfony

Le `WebhookProcessor` dispatch deux événements Symfony pour chaque webhook reçu :

| Événement | Classe | Moment |
|-----------|--------|--------|
| Réception | `WebhookReceivedEvent` | Avant les handlers |
| Traitement | `WebhookHandledEvent` | Après les handlers |

Les handlers intégrés dispatchent également des événements métier :

| Événement | Classe |
|-----------|--------|
| Abonnement créé | `SubscriptionCreatedEvent` |
| Abonnement mis à jour | `SubscriptionUpdatedEvent` |
| Abonnement supprimé | `SubscriptionDeletedEvent` |
| Paiement réussi | `PaymentSucceededEvent` |
| Paiement échoué | `PaymentFailedEvent` |

Exemple d'écoute :

```php
use CashierBundle\Event\SubscriptionCreatedEvent;
use Symfony\Component\EventDispatcher\Attribute\AsEventListener;

#[AsEventListener(event: SubscriptionCreatedEvent::class)]
final class OnSubscriptionCreated
{
    public function __invoke(SubscriptionCreatedEvent $event): void
    {
        $subscription = $event->getSubscription();
        // Envoyer un email, activer des features...
    }
}
```

## Traitement asynchrone

Pour traiter les webhooks de manière asynchrone avec Symfony Messenger, les messages suivants sont disponibles :

- `CancelSubscriptionMessage`
- `ProcessInvoiceMessage`
- `RetryPaymentMessage`
- `SyncCustomerDetailsMessage`
- `UpdateSubscriptionQuantityMessage`

Installez Messenger et configurez vos transports :

```bash
composer require symfony/messenger
```
